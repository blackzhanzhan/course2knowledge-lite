Technical Dossier

把在线视频课程编译成本地课程知识运行时。

Course2Knowledge Lite 不是 LMS、普通 RAG demo、笔记软件、视频下载器，也不是通用 Agent 框架。它把一门 B 站视频课当作证据源，经过字幕、视觉证据、中文讲义、知识原子和学习关口的编译，落到本地 SQLite 课程运行时，再由 Web Lite 课堂和可选 Hermes Lite 工具前台读取。

内核 SQLite Course Runtime

本地写入权威，保存课程、讲次、字幕、讲义、知识原子、关口、视觉证据、笔记、进度和聊天。

输入 Bilibili Course URL

合集、系列课、普通视频和多 P 视频都进入同一条公开导入管线。

前台 Web Lite / Hermes Lite

Web 负责可视化课堂，Hermes 负责可选工具调用。两者共享同一份课程 store。

Problem

视频课程的问题不是缺少总结，而是知识被困在时间轴里。

一门课程的知识散落在视频顺序、讲者语境、字幕片段、时间戳和画面证据中。只下载视频、转写字幕、让模型总结，或者把字幕直接丢进检索系统，都无法稳定回答：

  • 这门课有哪些讲？每一讲的来源和顺序是什么？
  • 每讲是否有可用字幕、讲义、知识原子、关口和视觉证据？
  • 回答问题时引用的是哪段字幕、哪个时间点、哪条课程证据？
  • 一次失败或半成品导入会不会覆盖已经可用的正式课程？
  • 网页前台和聊天前台是不是读取同一份本地状态？

Course2Knowledge Lite 的选择是先把课程编译为本地运行时，再在这个运行时上提供阅读、问答、聊天、笔记和状态展示。

Boundary

它不是什么：先把范式误会排除掉。

1. 课程是可编译对象

课程不是一组松散视频。系统把 lecture、字幕片段、讲义、知识原子、关口和视觉证据组织成稳定对象。

2. 证据链优先

回答问题必须尽量回到字幕片段、时间戳、来源 URL 或视觉证据。没有证据时应明确说明。

3. 前台只是投影

Web Lite 和 Hermes Lite 都读取同一个本地 store，不各自拥有业务状态。

Data Runtime

为什么是 SQLite：它是本地写入权威，不是临时缓存。

早期 JSON 适合作为 seed、fixture、迁移输入或调试导出，但正式运行时需要事务、索引、关系、状态查询和前台共享。Course2Knowledge Lite 因此把 SQLite 作为默认写入权威。

Import Pipeline

导入不是“抓一下字幕”，而是一条带质量保护的晋升管线。

  1. 课程展开

    把合集、系列课、普通视频、多 P 视频展开为有序 lecture，保留标题、BV/source URL 和 sequence。

  2. 字幕获取

    优先使用可用 B 站字幕；需要登录态时支持二维码登录、单次 cookie、remember-cookie 或本地环境变量。

  3. 证据归一化

    字幕被归一化为 timestamped transcript segments，成为讲义、问答和引用的证据底座。

  4. 课程编译

    每讲生成中文讲义、知识原子、学习关口和可选视觉关键帧。

  5. Readiness 检查

    检查 transcript、notes、atoms、gates 和必要元数据是否齐备。

  6. Promotion Gate

    只有质量可接受的候选课程才能合并或替换生产 SQLite store。

关键设计： 导入先写临时 store。缺字幕、缺讲义、缺知识原子、缺关口或探测性导入不会静默覆盖正式课程。

Frontdesks

Web 是课堂前台，Hermes 是可选工具前台。

Web Lite

Web Lite 是主要可视化前台。它不是控制台，而是轻量线上学校界面，围绕三个模块组织：互动模块、课程管理、课程笔记。

  • 互动模块：聊天、当前课程、当前视频、知识节点状态。
  • 课程管理：导入、删除、查看视频列表、查看 readiness。
  • 课程笔记：讲义、本地笔记、书签、阅读进度。

Hermes Lite

Hermes Lite 不是业务内核，也不是远程后端。它是可选工具前台，把同一 SQLite store 暴露给聊天界面。

  • 工具调用同一套 package API。
  • 不读取母项目私有 profile。
  • 不写 mastery、diagnosis、review queue。
  • 不依赖外部 Agent 框架。

Evaluation

评估不靠“效果很好”，而靠可验证指标。

导入完整性

课程是否能展开为正确 lecture 序列；每讲是否具备字幕覆盖；是否生成中文讲义、知识原子和关口。

晋升安全

临时 store 是否隔离；半成品是否阻断 promotion；探测导入是否不会覆盖正式课程。

前台一致性

Web Lite 与 Hermes Lite 是否读取同一 SQLite store；笔记、书签、阅读进度和聊天是否持久化。

发布安全

wheel 构建、单测、部署 smoke、前端语法检查和敏感路径扫描是否通过。

Release Boundary

发布包应该包含什么，不应该包含什么。

应该包含

  • Python package 与 wheel。
  • Web runtime assets。
  • Hermes Lite profile template 与 plugin。
  • 公开 docs 和安全 demo fixture。

不应该包含

  • 本地 SQLite 运行数据。
  • tmp/ 临时导入 store。
  • .codex/auth/ 或任何 B 站登录材料。
  • API key、真实 cookie、母项目私有运行时证据。

FAQ

常见疑问。

为什么不用纯 JSON？

JSON 适合 seed、fixture、迁移输入和调试导出，但正式运行时需要事务、查询、关系和多前台共享。SQLite 才是本地写入权威。

为什么不是普通 RAG demo？

RAG 只解释检索与回答。Course2Knowledge Lite 的核心是课程运行时：导入、证据、讲义、知识原子、关口、笔记、进度和聊天都落在同一 store。

为什么要 temp store 和 promotion gate？

因为真实课程导入会失败、缺字幕、缺关键帧或只做探测。临时 store 和 promotion gate 可以防止半成品覆盖正式课程。

Web Lite 和 Hermes Lite 谁是主？

都不是内核。内核是 SQLite 课程运行时。Web 是主要可视化课堂前台，Hermes 是可选工具前台。

Hermes 是不是系统内核？

不是。Hermes Lite 调用同一套 package API 和同一份 SQLite store。它是工具前台，不拥有业务状态。

项目是否依赖外部 Agent 框架？

不依赖。公共 Course2Knowledge Lite 的内核是本地 SQLite 课程运行时；Hermes Lite 只是可选工具前台。

cookie 存在哪里？

一次性 cookie 只用于当前导入。remember-cookie 只保存在本机 .codex/auth/，并被 git 忽略。文档、测试和发布包不能包含真实 cookie。

没有模型 key 能不能跑？

可以跑基础链路和确定性 fallback，但不能把 fallback 描述为模型质量等价。模型生成质量取决于本地配置的 provider。

为什么公共版没有 mastery / diagnosis / review queue？

这些属于母项目私有学习教练闭环。公共 Lite 的边界是课程知识运行时，不复制私有学习状态。

和母项目 Learning OS 是什么关系？

Course2Knowledge Lite 是从母项目能力中整理出的公共子项目。它保留课程到知识运行时的公开链路，剥离私有凭证、私有学习状态和生产聊天证据。

能不能接别的平台？

架构允许未来添加新的 source provider，但当前公共版本唯一真实平台适配器是 B 站。新增平台需要明确公共合同。

如何确认导入质量？

看 import status、transcript coverage、lecture dossier、knowledge atoms、gates、visual evidence 和 promotion 结果，而不是只看“有没有生成几句摘要”。
